# Program: Context, Hypothesis, and RECORD Lifecycle

## Objective

Fix three broken subsystems in the orchestrator (v0.8.60, 212 tests): context lifecycle enforcement, hypothesis persistence, and RECORD phase configurability. Forensic analysis of the README rewrite session revealed context entries stuck in "acknowledged" state with thin audit notes, hypotheses.yaml completely empty despite HYPOTHESIS phases executing, and RECORD phase forcing unnecessary journal/git operations.

## Forensic Evidence

From the README rewrite session (3 iterations, conversation 475137f7):
- `context.yaml`: entry `user_said_no_need_to_use_current_vers` stayed "acknowledged" with note `"seen by REVIEW"` through all phases - never transitioned to processed/dismissed even after being acted upon
- `hypotheses.yaml`: empty `{}` despite 2 HYPOTHESIS phases executing with 4-agent debates producing 3 hypotheses each. Root cause: `ACTION::HYPOTHESIS_AUTOWRITE` prompt tells subprocess to "Read the hypothesis output" but doesn't say WHERE - phase output is in `state.yaml:phase_outputs:HYPOTHESIS`, cold subprocess has no way to find it
- `failures.yaml`: entry `architecture_section_shows_auto_build` also stuck at "acknowledged" with thin note - same lifecycle problem as context
- RECORD phase gatekeeper rejected 3 times during planning iterations when there was nothing to commit, forcing dummy commits

## Pending Work Items

- **Context lifecycle enforcement** (high)
  - Scope: `orchestrator.py` cmd_start, _check_lifecycle_compliance, phases.yaml phase templates, NEXT template
  - Rich acknowledgment notes: the current hardcoded `{"acknowledged": "seen by PHASE"}` must be replaced with a generative note. The phase template prompt must instruct the agent to write a rich acknowledgment for each context entry: what action was taken, which agents referenced it, why it matters to this phase. The note is generated by the agent during the phase, not hardcoded in cmd_start
  - Guardian agent must verify that context acknowledgment notes are substantive (not just "seen by X") - add this to guardian checklist in phases.yaml
  - NEXT phase hard gate: all context entries must be either "dismissed" or "processed" before NEXT phase can complete. "acknowledged" is intermediate - it carries information about which phases saw the context, but it's not a resolution. The lifecycle check in `_check_lifecycle_compliance` currently only blocks on "new" entries at NEXT - it must also block on "acknowledged"
  - Transition enforcement: "acknowledged" entries that reach NEXT without being processed/dismissed should cause a hard failure with instructions to the agent to resolve each one
  - Gatekeeper prompt for NEXT must explicitly instruct: "For each acknowledged context entry, you MUST either mark it processed (if acted upon and added to PROGRAM/BENCHMARK) or dismissed (if not relevant or already incorporated)"
  - Acceptance: context entries cannot survive past NEXT in "acknowledged" state, notes contain rich generative audit information verified by guardian

- **Failure lifecycle alignment** (high)
  - Scope: `orchestrator.py` cmd_start failure acknowledgment, _check_lifecycle_compliance, phases.yaml phase templates
  - Same root cause as context: `cmd_start` hardcodes `{"acknowledged": "seen by PHASE"}` for failures too (L1866-1876). Forensic evidence: `failures.yaml` entry `architecture_section_shows_auto_build` stuck at "acknowledged" with thin note `"seen by REVIEW"` - never transitioned despite being resolved in iteration 2
  - Fix mirrors context exactly:
    1. Failure acknowledgment notes must be generative - phase template prompts agent to write rich note (what was investigated, what was found, was the failure addressed)
    2. cmd_start no longer hardcodes thin acknowledgment note for failures
    3. Guardian must verify failure acknowledgment notes are substantive
    4. `_check_lifecycle_compliance` blocks NEXT on "acknowledged" failures (same as context)
    5. NEXT gatekeeper instructs: "For each acknowledged failure, mark processed (if fixed with evidence) or dismissed (if not reproducible or out of scope)"
  - Acceptance: failures cannot be "acknowledged" at NEXT boundary, failure notes contain rich generative audit trail verified by guardian

- **Generative action execution mode** (high)
  - Scope: `model.py` ActionDef, `orchestrator.py` _run_auto_actions, phases.yaml action definitions
  - Root cause: all generative actions currently run as cold `claude -p` subprocesses via `_claude_evaluate`. The subprocess has no conversation context - it cannot find phase output, artifacts, or state. This is why hypotheses.yaml is empty: HYPOTHESIS_AUTOWRITE fires but the subprocess runs blind
  - Fix: add `execution` field to `ActionDef` dataclass with two values:
    - `"agent"` (default for generative) - runs as Agent tool call in the main process. Has full context, can read/write files, sees phase output
    - `"standalone"` - runs as `claude -p` subprocess (current behavior). For simple evaluations that don't need context
  - Each action in phases.yaml declares its execution mode explicitly:
    ```yaml
    ACTION::HYPOTHESIS_AUTOWRITE:
      cli_name: hypothesis_autowrite
      type: generative
      execution: agent          # runs as Agent in main process
      prompt: |
        Read phase output below and extract structured hypotheses.
        Write to {artifacts_dir}/hypotheses.yaml.
        ...
    ACTION::HYPOTHESIS_GC:
      cli_name: hypothesis_gc
      type: generative
      execution: agent
      prompt: ...
    ```
  - `_run_auto_actions` must check `action_def.execution`:
    - `"agent"`: resolve template variables ({phase_output}, {artifacts_dir}, {iteration}) into the prompt, then execute via Agent tool (or equivalent in-process mechanism)
    - `"standalone"`: resolve template variables, then call `_claude_evaluate` as before
  - Template variables for generative actions: `{phase_output}` (from state phase_outputs), `{artifacts_dir}` (DEFAULT_ARTIFACTS_DIR), `{iteration}` (current iteration number)
  - ACTION::HYPOTHESIS_AUTOWRITE prompt must include the phase output content and reference explicit path `{artifacts_dir}/hypotheses.yaml`
  - Acceptance: after HYPOTHESIS phase completes, hypotheses.yaml contains structured entries from the debate; execution mode is configurable per action in YAML

- **RECORD phase configurability** (high)
  - Scope: phases.yaml RECORD template and gatekeeper, orchestrator.py cmd_new, app.yaml
  - Current problem: RECORD template hardcodes journal append, git add, commit, push. Gatekeeper enforces all 3 even when there's nothing to commit (planning-only iterations)
  - Solution: two changes:
    1. Default RECORD behavior: only commit if there are actual code changes (evidence of `git diff --staged` or modified files required). If no code changes exist, RECORD passes with just an iteration summary. Gatekeeper must check: "Were there code changes in this iteration? If yes, journal + commit + push required. If no (planning/review only), iteration summary is sufficient."
    2. Add `--record-instructions` CLI flag on `orchestrate new` for custom instructions that augment the default template (e.g. "Update .claude/JOURNAL.md", "push to remote")
  - Store in state.yaml as `record_instructions`
  - The SKILL.md should document this flag
  - Acceptance: RECORD phase passes without journal/git when no code changes exist, requires full ceremony when code was modified, custom instructions augment the default

- **Hypothesis lifecycle alignment** (medium)
  - Scope: `orchestrator.py` _check_lifecycle_compliance
  - Currently HYPOTHESIS phase blocks on "new" hypotheses - good
  - Also need: hypotheses with status "new" must be resolved by NEXT phase (same pattern as context)
  - Acceptance: hypotheses follow same resolution enforcement at NEXT boundary

- **Clean on new must be thorough** (high)

  - `_CLEAN_PRESERVE` currently includes `{"context.yaml", "failures.yaml", "hypotheses.yaml"}` and `_CLEAN_PRESERVE_DIRS` includes `{"resources", "iterations"}` - these were added for `--continue` but they also apply to fresh `new`
  - Fix: the preserve sets must ONLY apply when `--continue` is used. Fresh `new` must delete everything except `resources/` directory
  - Acceptance: `orchestrate new` (without --continue) leaves only `resources/` in artifacts dir

- Rich notes must be generated by the agent, not hardcoded strings
- Template variables must use existing `_resolve_template` mechanism if possible
- No backward compatibility: old state.yaml files without new fields must crash with clear error instructing user to clean and restart
- No changes to FSM states - lifecycle enforcement is in _check_lifecycle_compliance, not FSM
