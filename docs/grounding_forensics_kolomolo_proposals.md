## Pain points observed

- Keyword collision: "run grounding" without a path matched Claude's world-model for "research" before matching the CLI. Cost: 4 WebSearch calls, ~60s, user interrupt `[abe54d1f:2136-2158]`. The skill description should explicitly claim the "grounding" keyword and the `validate-document` skill auto-trigger should fire earlier
- Two-step user correction: first "grounding must be run with process-document" was still too vague - Claude re-entered the `process-documents` skill and started reading its references instead of invoking `validate-document`. User re-issued as explicit `/document-processing:validate <path>` slash command `[abe54d1f:2208]` to force the right path. The skill routing on the keyword "grounding" alone is ambiguous between `process-documents` (which has a GROUNDING.md reference) and `validate-document` (the actual runner)
- Phantom `setup` subcommand `[abe54d1f:2213]`: SKILL.md instructs "Missing file → run `document-processing setup`" but the CLI in v1.0.38 only has `ground` and `ground-many`. Skill docs are ahead of the packaged CLI. cli.py now has `cmd_setup` defined (line 430), so this is a version-pinning issue - skill assumes a newer CLI than the user has installed
- Low lexical recall on transcripts: 15/36 = 42% CONFIRMED lexical-only for a 1715-word verbatim transcript source is surprisingly low given thresholds fuzzy=0.85 and bm25=0.5. Transcripts have high paraphrase distance because brief language is written ("Initial sizing targets approximately 10 developers") vs spoken ("those environments are going to be available to approximately 10 developers, at least for now"). The 0.85 fuzzy threshold essentially filters out every paraphrase
- Claim-extraction burden on the agent: the skill requires Claude to hand-author `claims.json` (36 rich paraphrases of the brief). No tooling assist. Claude had to read brief, mentally enumerate assertions, write JSON, then pass to CLI. This is the most expensive step token-wise and failure-prone (missed claims = unseen gaps)
- Auxiliary sources not text-searchable: `image.png` (deliverables spreadsheet) contributed 3 confirmed claims (C16, C30, C31) but the CLI silently scored them UNCONFIRMED because no text source. Manual audit recovered them. There's no way to pass "this claim is grounded in image X" to the tool
- Agent's first instinct is to override: Claude tried to absorb 58% UNCONFIRMED as "the tool underestimates, I'll do manual audit now" without re-asking the user. User slapped that back with explicit "you are not allowed to waiver this, you must check!!!" This is exactly what Rule 3 in SKILL.md exists for, but the skill allowed the agent to run a "manual audit" path without first offering semantic. Semantic consent was correctly gated (user had declined previously, stored in settings), but the agent's fallback was to skip the re-prompt and start generative grounding unprompted
- `validation-summary.md` gets re-written as telegram style per SKILL.md but the actual artefact (`validation-summary.md`) is 48-line flowing prose with headers and narrative - not telegram. The skill's template is aspirational; the agent emits the descriptive style that reads better for a human reviewer. Template compliance is zero
- Single-client validation was expensive and never repeated. 9 other clients have proposal drafts but no validation artefacts. The tool didn't scale to batch validation

## What worked well

- CLI `ground-many` output format is good: counts + per-claim ranked layer scores + passage quotations with line/column/paragraph locations. Claude used these pointers directly in the corrected report without re-reading source
- Semantic layer fixed the core lexical-recall problem: 15/36 -> 30/36 (+15 semantic confirms). Transcript paraphrase gap is exactly what ModernBERT is for
- Three-layer score disclosure (exact/fuzzy/bm25/semantic + agreement) let the agent build a defensible manual audit over top of the tool output instead of replacing it
- `opportunity_brief_corrected.md` was written cleanly with transcript-faithful range "12 to 15 months", explicit dev/test/staging vs dev/staging/production source-conflict note, C34 qualifier added. The workflow produced a shippable artefact
- SKILL.md's Rule 2 (contradiction is final) and Rule 3 (re-recommend semantic on struggle) were the rails that rescued the run when the agent drifted

## Concrete improvement recommendations

1. **Add explicit "grounding" keyword to `validate-document` skill description** (file: `document-processing/skills/validate-document/SKILL.md` frontmatter). Current description triggers on "validate/verify/check grounding/audit" - add "grounding" as a top-level keyword. Also add example triggers: "run grounding", "ground this document", "ground X against Y". Effort: small. Impact: high - eliminates the keyword-collision that cost 4 WebSearch calls
2. **Fix `setup` subcommand version gap** (file: `src/stellars_claude_code_plugins/document_processing/cli.py` + `src/stellars_claude_code_plugins/__init__.py` version). Either bump PyPI release so installed CLI actually has `cmd_setup`, or add a version check in SKILL.md - "requires document-processing >= 1.4.0". At minimum, make the CLI fail gracefully for the `setup` command on old versions with a clear "upgrade: pip install --upgrade stellars-claude-code-plugins" message. Effort: small. Impact: medium - eliminates the misleading error mid-workflow
3. **Add `clients/*/` batch validation command** (file: new `document-processing/commands/validate-all.md` + skill support). 9 of 10 clients were never validated. Pattern is uniform: each client has `resources/<source>` and one or more `<client>_brief.md` / `opportunity_brief.md`. Command should accept a glob, loop, write `validation/` per client, emit roll-up report. Effort: medium. Impact: high for kolomolo - unlocks the 90% backlog
4. **Add `source_map.yaml` convention** (file: new `document-processing/skills/validate-document/references/SOURCE_MAP.md` + parser support). For each proposal doc, declare: `{doc: opportunity_brief.md, sources: [resources/Actone.txt, resources/image.png], image_sources: [resources/image.png]}`. The skill reads this first instead of asking the agent to discover mapping every run. Covers the image.png case by letting claims declare an image-only source - agent then knows to manually verify rather than mark UNCONFIRMED. Effort: medium. Impact: high - eliminates per-run discovery and handles non-text sources cleanly
5. **Lower default fuzzy threshold to 0.75 for transcripts, or add `--source-kind transcript` profile** (file: `src/stellars_claude_code_plugins/config_document_processing.yaml`). Transcripts paraphrase-distance from written briefs is consistently above the 0.85 cutoff - measured 15 lexical confirms on 36 claims where a further 15 were rescued by semantic. For transcript sources, 0.75 would likely catch most of the rescued claims without requiring semantic. Add a preset in config: `transcript_profile: {fuzzy_threshold: 0.75, bm25_threshold: 0.4}`. Effort: small. Impact: medium - reduces semantic dependence when it isn't installed
6. **Auto-extract claims from the target document** (file: new `src/stellars_claude_code_plugins/document_processing/claims_extractor.py` + `document-processing extract-claims` subcommand). Current flow: agent reads brief, writes claims.json by hand (36 entries, ~2000 tokens). A claim extractor using sentence splitting + noun-phrase filtering + heuristic "factual assertion" detection would cut this cost and prevent missed claims. CLI: `document-processing extract-claims --doc opportunity_brief.md --output validation/claims.json`. Agent still reviews/edits, but starts from 80% coverage. Effort: large. Impact: high - claim extraction is the highest-token step in the workflow
7. **Add tool-side "numeric tightening" detection** (file: `src/stellars_claude_code_plugins/document_processing/grounding.py` around `_compute_agreement_score`). C23 "12 to 14 months" vs source "12 to 24 months" was found by manual audit - tool saw bm25 match and returned CONFIRMED. Extend numeric_mismatches to detect range narrowing: if claim has `N to M` and best passage has `N to K` where K > M, flag as TIGHTENED (new match_type between CONFIRMED and CONTRADICTED). Effort: medium. Impact: medium - catches a whole class of subtle misalignments that contract-sensitive docs care about
8. **Fix telegram-style template enforcement in SKILL.md** (file: `document-processing/skills/validate-document/SKILL.md`). The generated `validation-summary.md` was flowing prose with 48 lines of narrative; the SKILL template is strict telegram-style. Agents ignore the template when they judge prose is more informative. Either tighten the skill instruction (add "REJECT your draft if it exceeds 25 lines or contains sentences longer than 15 words") or drop the telegram aspiration and match the prose reality. Effort: small. Impact: low-medium - style/compliance consistency
9. **Add `validation/` auto-discovery for re-runs** (file: `document-processing/skills/validate-document/SKILL.md` Phase 0). If `validation/` exists for the doc, load `criteria.md` + `claims.json` rather than re-gathering. Supports the "fix and re-validate" loop after corrections. Effort: small. Impact: medium - makes the corrections pass cheap
10. **Handle image/binary sources explicitly** (file: `src/stellars_claude_code_plugins/document_processing/grounding.py` + skill). Current behavior: image sources silently score 0. Three options: (a) accept `--image-source` flag that marks claims as image-grounded and skips lexical scoring; (b) OCR image via pytesseract and treat as text; (c) require the agent to pre-extract image content to a sidecar `.md`. Option (a) is lowest-effort and matches what actone's manual audit did anyway. Effort: small for (a), large for (b). Impact: medium - 3 of 36 claims in the actone run

## Explicit non-recommendations

- **Don't add a "default accept" mode for low grounding scores**: the session proved that the agent already overreaches (Claude's "the tool underestimates" retreat was exactly the failure mode Rule 3 guards against). Rule 3 (ask user) is working; don't weaken it
- **Don't pre-install the semantic extra by default**: 150MB first-run model download is the right thing to gate behind consent, per SKILL.md. The user explicitly approved it mid-session when needed and the flow worked
- **Don't auto-promote `opportunity_brief_corrected.md` over the original**: user explicitly wanted the original preserved; current Phase 5 behavior is correct ("A corrected copy will be written... decide whether to promote")
- **Don't add more thresholds or layers**: already 4 layers (exact/fuzzy/bm25/semantic) + agreement_score + entity penalty + numeric mismatch detection. Any additional layer adds explanation cost without clear recall gain - the actone case went from 42% to 97% via existing machinery, the rescues were in claim extraction, source-mapping, and image handling, not layer count
